러스트에서 범용적인 상황에서는 잘 쓰이지 않는 메모리 pin에 대해서 알아보자.

일반적으로 사용할 경우 별로 접할 수 있는 기회가 없다. 공식 문서에서도 케이스가 따로 없는 상황이다.

보통 이 `Pin`을 볼 케이스는 두가지인데

1. 비동기
2. 극단적인 최적화

정도가 되겠다.

우선 비동기부터 빠르게 알아보자.

## 비동기에서의 `Pin`

이건 좀 복잡하다. 일단 러스트의 기본적인 비동기 동작부터 이해해야한다. 예를 들어 다음과 같은 비동기 함수가 있다고 가정하자:

```rust
async fn foo() {
    let mut buffer: [u8, 2048];
    let ptr = &mut buffer;
    
    some_io_action(ptr).await;
    
    println!("bar")
}
```

비동기 함수 내부를 보면 자기 참조가 있다.

그럼 컴파일러가 상태머신을 다음과 같이 정의할 것이다 (실제로는 더 복잡하다, pseudo code인 걸 참고바람):

```rust
enum FooStateMachine {
    Start,
    // await 지점의 상태
    YieldedAtSomeIo {
        buffer: [0u8; 1024],
        ptr: *mut [0u8; 1024], // buffer를 가리키는 자기 참조!
    },
    Done,
}
```

만약 foo 비동기 상태가 이동되면 `buffer`의 메모리 주소가 재배치될 수 있을 것이다. 그 상태에서 `ptr`을 활용한 어떤 동작을 수행한다면 UB가 될 것이다.

예를 들어서 `Vec` 내에 `Future`를 넣는다고 가정해보자.

```rust
let mut foo = Vec::new();

// bar 비동기 작업을 넣음
foo.push(bar());

// 첫번째 비동기 작업 실행, 내부적으로 참조 포인터 생성
foo[0].poll();

// 새로운 작업이 들어와서 foo 작업 큐에 추가됨
foo.push(bar());

foo[0].poll();
```

위 코드에서 만약에 `poll()`을 시도하면 UB가 된다. 벡터는 capacity가 차면 메모리를 재배치하기 때문이다.

그럼 단순히 Box로 감싸면 원본 스택은 재배치가 안 되니까 문제가 안되지 않나? 라고 하면 사실 맞다.

근데 Box같은 경우는 내부 `&mut T`를 제공하기 때문에 safe 보장이 되지 않는다. 의도적으로 `mem::swap` 을 때려버리면 (그럴 가능성이 있을까 싶지만) 메모리 참조 시에 프로그램 전체가 터질 것이다.

그래서 러스트는 `Pin<T>` 구조체를 제공해준다. 부가적인 기능이 있는게 아니라 그냥 `&mut T` 제공 안해주는 포인터 래퍼다. 그래서 `Future` 트레잇을 보면 `poll`의 인자가 `Pin<&mut Self>` 다.

## 극단적인 최적화

이제 극단적인 최적화 케이스에서의 pin이 어떻게 사용되는지 알아보자.

`Linked List`를 빡세게 최적화한 침투형 연결 리스트(`Intrusive Linked List`)가 대표적인 예시 중에 하나다.

일반 `Linked List`는 보통 다음과 같이 구현된다.

```rust
struct Node<T> {
    data: T,
    prev: *mut Node<T>,
    next: *mut Node<T>,
}
```

노드가 하나 추가될 때마다 힙 할당이 되고 해당 포인터를 노드가 가져가는 식이다.

이러면 매번 힙 메모리 할당 비용이 들어갈 것이다. 비용적으로 이런게 꺼려지면 그냥 노드에 바로 스택 참조를 떄리는 침투형 연결 리스트를 사용할 수 있다.

코드는 다음과 같다:

```rust
use std::marker::PhantomPinned;
use std::pin::Pin;

struct ListLink {
    prev: *mut ListLink,
    next: *mut ListLink,
}

struct Order {
    price: u64,
    link: ListLink,
    
    // !Unpin (stable에는 negative impl이 안됨)
    _marker: PhantomPinned, 
}
```

지금 보면 `PhantomPinned`라는 타입이 있는데 이건 `!Unpin` trait을 바운드한 팬텀 타입이다. safe rust에서는 nagative impl이 안돼서 저렇게 쓴다.

`Unpin` 트레잇은 Pin을 해제할 수 있게 해주는 트레잇인데, `Pin<T>::get_mut`으로 내부 뮤테이션을 가져올 수 있냐 없냐를 결정하는 트레잇이다.

기본적으로 `Auto Impl`이라 기본 타입에 다 자동 구현된다. 다만 위에서 설명한 `PhantomPinned`랑 `Future` 트레잇에는 구현되지 않는다.

이걸 negative impl하게 하면 절대 핀 내부의 값을 safe rust에서는 가져올 수 없게 만든다.

저기서 단순히 `Order` 구조체에 함수를 추가해서 `self`를 `Pin<Self>`로 받고 내부에서 잘잘 처리하면 될 것 같다고 생각할 수 있는데, 지금 보면 `ListLink`는 구조체의 주소가 아니라 내부 필드를 참조한다. 그래서 원본 `Order` 구조체를 가져오려면 내부에서 포인터 역산을 때려야한다.

다행히도 러스트 1.77 이후 버전부터 포인터 역산 매크로를 지원해준다. 일일이 타입 바이트를 계산하지 않아도 된다.

```rust
impl ListLink {
    unsafe fn get_parent(&self) -> *mut Order {
        // 포인터 주소 구하기
        let link_ptr = self as *const ListLink as *const u8;
        
        // offset_of! 매크로를 사용해서 Order 내의 link 필드 오프셋 구하기
        let offset = offset_of!(Order, link);
        
        // link 주소에서 오프셋만큼 빼서 본체의 시작 주소로 이동
        let parent_ptr = link_ptr.sub(offset);
        
        // 원본 Order 캐스팅
        parent_ptr as *mut eOrder
    }
}
```

unsafe하긴 한데 `Order` 내에 `ListLink`가 있다고 개발자가 책임을 보장한다면 저렇게 사용할 수 있다.

그리고 또 개발자가 신경써줘야할 게 좀 많다. 

### Pin 투영 (Pin projection)

어떤 `Pin<T>`에 대해서 필드를 가져오는 헬퍼 함수를 작성할 때 노출되는 필드들은 모두 `Pin<T>`이거나 아니어야 한다.

즉, 어떤 함수에선 `Pin<F>` 고, 어떤 함수에선 `F`이면 안된다는 말이다. 왜냐면 `Pin`은 단순 wrapper 타입이라 내부 값이 어떻게 되든 보장하지 않기에, 개발자가 이를 보장해야한다.

이걸 두가지의 경우로 나눈다.

1. Structural Pinning
- 내부 필드 재배치를 허용하지 않는 걸 보장한다.

2. Non-Structural Pinning
- 내부 필드 재배치를 허용한다.

제일 중요한 것은 **보장의 책임이 전부 개발자에게** 있다. 혹시라도 코드를 외부로 제공한다면 꼼꼼히 체크해야한다. 어떤 필드는 재배치를 허용하고, 어떤 필드는 허용하지 않으면 UB가 일어날 수도 있다.

### Drop 보장성

Drop 시에 `&mut T`를 주기 때문에 잘 처리해야한다. 예를 들어서 Structural한 필드를 재배치하거나 하면 UB이다. 보장되지 않기 때문이다.
또한, `#[repr(packed)]` align 매크로도 사용하면 안된다. 자동 재배치르 UB가 생길 수 있다.
